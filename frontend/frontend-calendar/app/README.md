# Финансовый модуль — фронтенд (React)

Реализация всех экранов из макета «Макеты.pdf», подключённая к реальному API
(`openapi.yaml`, лежит в корне проекта для справки).

## Стек
React 18 + React Router 6 + axios, сборка на Vite.

## Запуск
```bash
npm install
cp .env.example .env   # по умолчанию http://localhost:8000/api — поправьте при необходимости
npm run dev
```

## Соответствие разделов приложения и API

| Раздел в приложении | URL в приложении | Ресурс в API | Возможности |
|---|---|---|---|
| Авторизация | `/login` | `POST /login`, `GET /me`, `POST /logout` | — |
| Платёжный календарь | `/calendar` | `GET /calendar` | фильтр по счёту, подсветка кассовых разрывов (`is_cash_gap`) |
| Заявки | `/requests` | `/payments` | CRUD + workflow: отправить на согласование, согласовать/отклонить, вернуть в черновик, отметить оплаченной |
| Поступления | `/receipts` | `/incomes` | CRUD |
| Шаблоны | `/templates` | `/recurring-templates` | CRUD + кнопка «Сгенерировать» (`/generate`) |
| Реестры | `/registries` | `/registries` | CRUD + «Заполнить» (`/attach`) и «Экспорт» (Excel) |
| Отчёты | `/reports` | `/reports/balances`, `/reports/cash-gaps`, `/reports/plan-fact` + `/export/*` | 3 вкладки, каждая с Excel-экспортом |
| Панель администратора → Пользователи | `/admin/users` | `/admin/users` | CRUD (роли — мультивыбор) |
| Панель администратора → Контрагенты | `/admin/counterparties` | `/counterparties` (в API верхнеуровневый путь, не `/admin/*`) | CRUD |
| Панель администратора → Счета | `/admin/accounts` | `/accounts` | CRUD |
| Панель администратора → Статьи | `/admin/articles` | `/items` | **только список + создание** — в API нет `GET/PUT/DELETE /items/{id}` |
| Журнал аудита | `/audit-log` | — | **в API такого эндпоинта нет** — страница честно сообщает об этом |

## Важные особенности реализации

- **Суммы в копейках.** API хранит и принимает все суммы (`amount`, `initial_balance`) как целые числа в копейках. В формах и таблицах показываются рубли — конвертация в обе стороны в `src/utils/money.js` и `src/pages/EntityFormPage.jsx` (через `moneyFields` в конфиге сущности).
- **Роли пользователей** (`initiator`, `treasurer`, `manager`, `admin`) не возвращаются в `GET /admin/users/{id}` (только при логине/`/me`) — при редактировании существующего пользователя чекбоксы ролей будут пустыми, нужно выставить заново перед сохранением.
- **ID валюты** (`currency_id` у счетов) — в API нет справочника валют, поле — обычное число с подсказкой в форме.
- **Экспорт в Excel** качается через axios с `responseType: 'blob'` (см. `src/utils/download.js`), а не прямой ссылкой — потому что эндпоинты требуют Bearer-токен.
- **routePath vs basePath.** В `src/config/entities.js` у каждой сущности два пути: `routePath` — URL внутри приложения (соответствует структуре сайдбара из макета) и `basePath` — реальный путь в API. Разделено осознанно: например, «Контрагенты» открываются на `/admin/counterparties`, а в API это `/counterparties` без префикса `/admin`.

## Что не реализовано (осознанно, не входило в объём макета)
- Импорт заявок/поступлений из Excel (`/payments/import`, `/incomes/import`)
- Курсы валют (`/rates`, `/rates/update`)
- Перенос даты/счёта заявки (`/reschedule`, `/change-account`)
- Сброс пароля пользователя (`/admin/users/{id}/reset-password`)
- Пагинация и серверные фильтры в списках (списки грузятся целиком одним запросом)

Всё перечисленное подключается по тому же шаблону, что и остальные экраны —
API-функция в `src/api/*.js` + кнопка/поле в соответствующей странице.

## Структура проекта

```
openapi.yaml      Спецификация API (для справки)
src/
  api/            HTTP-клиент и запросы к API (auth, resource, payments, registries, templates, reports, calendar)
  components/     Sidebar, Layout, DataTable, EntityForm — переиспользуемые UI-компоненты
  config/         entities.js — конфигурация всех CRUD-разделов (поля форм, колонки, пути)
  context/        AuthContext — авторизация, роли пользователя
  pages/          Страницы приложения (+ requests/registries/templates — специфичные workflow-кнопки)
  styles/         global.css — стили в палитре макета (зелёный/красный)
  utils/          money.js (копейки в рубли), download.js (скачивание Excel)
```

## Аудит по документам ТЗ (обновление)

После сверки с полным пакетом документов (`Модель_данных.docx`, `Памятка_роли_и_обязанности.docx`, `Пользовательские_сценарии.docx`, `ТЗ-03_Платёжный_календарь.docx`, `Ответы_от_заказчика.docx`, `Правила.docx`) внесены исправления:

### Исправлено
- **Drag-and-drop перенос заявки в календаре** (критерий приёмки №4 ТЗ-03). Заявки отображаются в ячейках дня жёлтыми плашками, их можно перетащить на другой день — вызывается `POST /payments/{id}/reschedule`, предупреждение о кассовом разрыве (`response.warning`) показывается пользователю после переноса.
- **Полные фильтры календаря**: счёт, статья, контрагент, статус (было — только счёт).
- **Импорт из Excel** для заявок и поступлений (`/payments/import`, `/incomes/import`) — кнопка и модалка с результатом импорта и списком ошибок по строкам.
- **Переключатель «Все заявки / Мои заявки»** на странице заявок (сценарий 2.9). Фильтрация клиентская, по `created_by` — API не документирует серверный параметр для этого.
- **Ролевая видимость пунктов меню** (Правила п.1.5): Календарь/Реестры/Отчёты скрыты от чистого Инициатора; Панель администратора — только Администратору; Журнал аудита — Администратору и Руководителю.
- **Редирект после входа по роли** (сценарий 1.1): «Заявки» для Инициатора, «Календарь» — для остальных.
- **Обязательные поля** у Счетов и Контрагентов приведены в соответствие с Правила п.2.1/2.2 (все реквизитные поля обязательны, не только «Название»).
- **Фильтрация статей по типу**: в заявках доступны только статьи типа `expense`, в поступлениях — только `income`, в шаблонах — динамически в зависимости от выбранного типа шаблона (Правила п.3.1/4.1/6.1).
- **Валюта счёта** — заменил произвольный ввод ID на выпадающий список (RUB/USD/EUR/AMD по «Правила» п.1.4), но соответствие ID⇄код валюты в реальной БД нужно проверить и поправить при необходимости.

### Известные ограничения — не в зоне фронтенда (см. пункт про API-пробелы выше)
- **Журнал аудита** — в API нет эндпоинта, хотя в модели данных таблица `audit_log` есть и подробно описана в сценариях (4.4, 5.4). Нужно с бэкенд-разработчиком.
- **Хранение истории отчётов** — Правила п.8.2 требуют хранить старые отчёты, но в API нет `GET /reports` (списка сохранённых), только генерация «на лету».
- **Экспорт в PDF** — Правила/сценарии упоминают Excel и PDF для реестров и отчётов, в API есть только Excel.
- **«Перенести все последующие платежи» для регулярного шаблона** (сценарий 3.2, Ответы заказчика) — `PaymentReschedule` в API принимает только `new_date`, без признака "применить ко всем следующим". Функционально при перетаскивании переносится только один платёж.
- **Конвертация валюты при переносе на счёт в другой валюте** (сценарий 6.3) — эндпоинт `POST /payments/{id}/change-account` в API есть, но диалог подтверждения конвертации в интерфейсе пока не добавлен (нет отдельного экрана для этого действия, есть только программный вызов в `src/api/payments.js`).
- **Утверждение реестра / отметка оплаты** — в API нет отдельных `approve`/`mark-as-paid` для реестров, это делается через смену `status` в форме редактирования реестра (использует общий `PUT /registries/{id}`), что соответствует бизнес-логике, но не выделено отдельными кнопками с понятными подписями «Утвердить»/«Подтвердить оплату», как показано в макете.

## Дополнение аудита (раунд 2)

- **Важно: API не возвращает `created_by` у заявки** (хотя поле есть в модели данных БД). Из-за этого:
  - переключатель «Мои заявки» теперь честно **отключается** с иконкой ⚠ и подсказкой, если поле отсутствует в ответе, вместо того чтобы молча показывать пустой список;
  - кнопки «Отправить на согласование» / «В черновик» ограничены только по роли (Инициатор/Администратор), а не по авторству конкретной заявки — так как отличить «свою» заявку от чужой на фронте нечем.
- Добавлена кнопка **«Отправить повторно»** для отклонённых заявок (Правила п.3.4) — раньше отклонённые заявки вообще не имели действий.
- Реестры: вместо общей кнопки «Редактировать» добавлены явные **«Утвердить»** (Руководитель/Админ, draft→sent_to_bank) и **«Подтвердить оплату»** (Казначей/Админ, sent_to_bank→paid) с диалогом подтверждения — как в макете.
- Статус реестра скрыт из формы создания — по Правила реестр всегда создаётся как черновик, статус меняется только через действия.
- Приоритет в шаблонах обязателен только для типа «Платёж» (Правила п.6.1) — динамическая валидация.
- Добавлен перенос заявки на другой счёт («На другой счёт» — доступно Казначею/Админу для любых заявок кроме оплаченных) с диалогом подтверждения конвертации валюты, если у целевого счёта другая валюта (сценарий 6.3, Правила п.5.6).

## Дополнение аудита (раунд 3) — бекенд закрыл несколько пробелов

Бекендер добавил недостающие эндпоинты (они не попадали в исходный `openapi.yaml`, так как сгенерированы отдельно). Подключил их по-настоящему:

- **`created_by` теперь есть в `Payment`** → переключатель «Мои заявки» и кнопки «Отправить на согласование»/«В черновик» теперь проверяют реальное авторство заявки (`payment.created_by === user.id`), а не только роль.
- **Журнал аудита подключён по-настоящему**: `GET /audit/logs` (список), `GET /audit/logs/{id}`, `GET /audit/payment/{paymentId}` (история конкретной заявки — доступна с новой кнопки «История» у заявок, видна admin/manager).
- **История отчётов**: `GET /reports/history` — новая вкладка «История отчётов» на странице отчётов.
- **Статьи (`/items`) — полноценный CRUD**: добавление/редактирование/удаление, доступно только администратору (кнопки скрыты для остальных ролей на фронте, бекенд тоже ограничивает).

### Что осталось уточнить у бекендера (сделал по обоснованным предположениям, раз точной схемы ответа не было)
- Точный формат ответа `GET /audit/logs` и `GET /reports/history` (обёртка `{items:[...]}` / `{data:[...]}` / просто массив, названия полей типа `user_name` vs вложенный `user.name`) — код обрабатывает несколько вариантов защитно, но стоит свериться.
- **Скачивание конкретного сохранённого отчёта** из истории — если у записи нет прямого `file_url`, в таблице стоит пометка «нет ссылки», потому что эндпоинт вида `GET /reports/history/{id}/download` бекендером пока не подтверждён. Как только пришлёт — подключу мгновенно.

## Дополнение аудита (раунд 4)

- Уточнено: ошибка `type_text` в шаблонах (`recurring_templates`) — это НЕ про отсутствие поля `name` (оно у шаблонов было в форме с самого начала), а отдельный баг на бекенде (модель пытается сохранить вычисляемые поля вроде `type_text`/`amount_formatted` в колонки, которых нет в таблице).
- По аналогии с тем, что случилось у Заявок/Поступлений (поле `name` требуется бекендом, но не задокументировано в `RegistryCreate`/`PaymentCreate`/`IncomeCreate` схемах `openapi.yaml`), проактивно добавил поле **«Название»** и в Реестры — в макете оно тоже есть («Новый реестр» → «Название:»), а в присланной спецификации `RegistryCreate` его нет.
